# Product Requirements Document (PRD)

## 1. Введение и цели проекта
**Описание:**
CARCARE — современное веб- и мобильное приложение для учёта расходов, заправок, техобслуживания и штрафов для частных автолюбителей.
**Цели:**
 Мобильная и веб-версия: PWA на Svelte 5.x, SvelteKit 2.x, Vite 8.x, Node.js 20.19+, генерация apk через TWA/Bubblewrap
    - Первый этап: генерация apk для ручной установки и тестирования (без публикации)
    - Второй этап: публикация apk в RuStore (приоритетно)
    - Третий этап: публикация apk в Google Play (по необходимости)
    - Все этапы сопровождает javascript-expert
    - Тестирование: Vitest 4.x (unit/integration), Playwright (e2e)
    - @sveltejs/vite-plugin-svelte
- Основная аудитория: частные автолюбители (B2C)
- Поддержка всех устройств: мобильные, десктоп, планшеты
- Языки: русский и английский
**Ответственный агент:** product-manager

## 3. Пользовательские сценарии и ключевые фичи
- Регистрация/авторизация (email, Яндекс, Google)
- Профиль пользователя (аватар, имя, авто, галерея)
- Разделы: Главная (виджеты), Топливо, Техосмотр, Штрафы, Отчёты
- Платные функции: добавление второго и последующих авто

### Детализация пользовательских сценариев и ключевых фич

**Основные сценарии:**
1. Регистрация и авторизация (email, Яндекс, Google)
   - Ввод email/пароля, OAuth
   - Подтверждение email
   - Критерии: успешная регистрация, валидация, безопасность
2. Управление профилем
   - Редактирование имени, аватара, темы, часового пояса
   - Добавление/редактирование автомобиля (марка, год, фото, галерея)
   - Критерии: все поля валидируются, фото до 2 МБ, не более 3 фото в галерее
3. Главная страница (виджеты)
   - Отображение плиток с отчётами и действиями
   - Кастомизация виджетов
   - Критерии: быстрый отклик, актуальные данные, настройка порядка
4. Раздел "Топливо"
   - Список заправок (сортировка по дате)
   - Создание новой заправки (адрес, объём, тип топлива, цена, пробег, авто, дата/время)
   - Автоматическое подгружение ранее введённых сущностей
   - Критерии: автозаполнение, валидация, сохранение истории
5. Раздел "Техосмотр"
   - Ведение расходов по ТО (статья, цена, количество, дата, авто, пробег)
   - Критерии: корректность расчётов, фильтрация по авто/дате
6. Раздел "Штрафы"
   - Ввод оплаченных штрафов (статья, сумма, авто, дата)
   - Критерии: возможность расширения для интеграции с Госуслугами
7. Раздел "Отчёты"
   - Формирование отчётов по заправкам, пробегу, ТО, штрафам
   - Фильтры: период, авто, тип расхода
   - Визуализация: таблицы, графики, круговые диаграммы
   - Критерии: переключение между видами, экспорт, актуальность
8. Платные функции
   - Добавление второго и последующих авто
   - Критерии: ограничение для бесплатных, оплата, активация

**Чек-лист приёмки сценариев:**
- Все поля валидируются и имеют ограничения
- Ошибки отображаются пользователю понятно
- Все действия логируются (для аудита)
- UI/UX соответствует гайдлайнам (ui-ux-pro-max)
- Все сценарии покрыты тестами (unit/integration/e2e)

**Ответственные агенты:**
- Проработка сценариев: product-manager
- UI/UX: ui-ux-pro-max
- Имплементация: golang-expert, javascript-expert
- Тестирование: python-expert, golang-expert, javascript-expert

## 4. Технические требования и архитектура
- Backend: Go, DDD, TDD, Clean Architecture
- Frontend: Svelte 4 (стабильная версия), SvelteKit, Vite, SPA, стеклянный/минималистичный дизайн, тёмная/светлая тема
- Мобильная версия: выбор подхода — javascript-expert
- Docker, Docker Compose для деплоя
- Безопасность: rate limiting, защита API, соответствие ФЗ-152 и GDPR
- Масштабируемость: вертикальная с возможностью горизонтальной
**Ответственные агенты:** architecture-expert, golang-expert, javascript-expert, docker-expert, security-expert

### Детализация архитектуры и технических требований

**Архитектурный стиль:**
- Clean Architecture (основа)
- Hexagonal Architecture (для интеграций)
- Domain-Driven Design (DDD) для бизнес-логики

**Слои системы:**
1. Entities (ядро бизнес-моделей)
2. Use Cases (правила приложения, сценарии)
3. Interface Adapters (REST-контроллеры, presenters, gateways)
4. Frameworks & Drivers (Svelte SPA, PostgreSQL, RabbitMQ, внешние API)

**Ключевые принципы:**
- Все зависимости направлены внутрь (ядро не зависит от инфраструктуры)
- Бизнес-логика не зависит от UI, БД, внешних сервисов
- Внедрение зависимостей через интерфейсы (инверсия зависимостей)
- Возможность unit-тестирования бизнес-логики без запуска инфраструктуры
- Чёткое разделение bounded contexts (например: учёт топлива, штрафы, ТО)
- Использование ubiquitous language для моделей и API

**Масштабируемость:**
- Вертикальная (увеличение ресурсов)
- Возможность горизонтального масштабирования (репликация PostgreSQL, кластеризация RabbitMQ, stateless backend)

**Тестируемость:**
- Покрытие unit/integration тестами всех слоёв
- Моки/стабы для внешних сервисов

**Расширяемость:**
- Добавление новых API/интеграций без изменения ядра
- Возможность внедрения новых bounded contexts

**Критерии приёмки архитектуры:**
- Соблюдение слоёв и зависимостей (code review architecture-expert)
- Документированная схема слоёв и взаимодействий
- Наличие тестов для бизнес-логики
- Возможность запуска приложения с mock-инфраструктурой
- Поддержка расширения API и интеграций

**Ответственные агенты:**
- Проектирование архитектуры: architecture-expert
- Code review архитектуры: architecture-expert, security-expert
- Имплементация слоёв: golang-expert, javascript-expert
- Документирование: product-manager

## 5. UI/UX требования и дизайн
- Стиль: glassmorphism/minimalism, адаптивность, accessibility
- Виджеты на главной, переключение тем
- Максимальный размер фото: 2 МБ, форматы JPEG/PNG
**Ответственный агент:** javascript-expert (использует skill ui-ux-pro-max)

### Детализация UI/UX требований и дизайна

**Основные стили и подходы:**
- Glassmorphism/minimalism (современный, лёгкий, прозрачные элементы)
- Переключение между тёмной и светлой темой
- Адаптивность: mobile-first, поддержка всех устройств
- Максимальный размер фото: 2 МБ, форматы JPEG/PNG

**UI/UX гайдлайны (приоритеты):**
1. Accessibility (контраст, alt-тексты, навигация с клавиатуры, aria-labels)
2. Touch & Interaction (размеры элементов, отступы, обратная связь)
3. Performance (lazy loading, оптимизация изображений, отсутствие layout shift)
4. Style Selection (консистентность, SVG-иконки, отсутствие смешения стилей)
5. Layout & Responsive (мобильные брейкпоинты, отсутствие горизонтального скролла)
6. Typography & Color (базовый размер 16px, line-height 1.5, семантические цвета)
7. Animation (длительность 150–300мс, смысловая анимация, поддержка reduced-motion)

**UI элементы:**
- Виджеты (плитки) на главной: настраиваемые, drag&drop, быстрый доступ к отчётам
- Формы: валидация, автозаполнение, понятные ошибки
- Таблицы и графики: переключение между режимами, экспорт
- Навигация: sidebar/menu, быстрый доступ к разделам

**Критерии приёмки UI/UX:**
- Соответствие гайдлайнам ui-ux-pro-max
- Проверка accessibility (ручная и автоматизированная)
- Адаптивность на всех устройствах
- Быстрый отклик интерфейса (<1 сек)
- Понятные ошибки и обратная связь
- Все основные сценарии покрыты UI-тестами

**Ответственные агенты:**
- Проработка и контроль UI/UX: javascript-expert (skill ui-ux-pro-max)
- Имплементация: javascript-expert
- Тестирование: python-expert, javascript-expert

## 8. Тестирование и качество
- Unit, integration, e2e тесты, TDD
- Критерии приёмки: покрытие тестами, review
**Ответственные агенты:** golang-expert, javascript-expert, python-expert

### Детализация тестирования и качества

**Виды тестов:**
- Unit-тесты: изолированная проверка функций, бизнес-логики (Go, JS, Python)
- Integration-тесты: взаимодействие компонентов, API, БД, очереди
- E2E (end-to-end): пользовательские сценарии через UI (автоматизация)
- Performance-тесты: отклик, нагрузка, стресс

**Подходы:**
- Test-Driven Development (TDD) для бизнес-логики и ключевых сценариев
- Использование моков/стабов для внешних сервисов и интеграций
- Изоляция тестов, отсутствие shared state
- Покрытие тестами всех критичных путей (минимум 80% для ядра)
- Автоматический запуск тестов в CI/CD (GitHub Actions)

**Критерии приёмки тестирования:**
- Все основные сценарии покрыты unit/integration/e2e тестами
- Тесты проходят стабильно, без флаки
- Покрытие ядра >80%, UI/интеграций >60%
- Все внешние интеграции замокированы в тестах
- Performance-тесты для ключевых API и UI
- Результаты тестов доступны в CI/CD pipeline

**Ответственные агенты:**
- Стратегия тестирования: product-manager, python-expert
- Имплементация тестов: golang-expert, javascript-expert, python-expert
- Code review тестов: code-review-excellence

## 9. CI/CD и DevOps
- Автоматизация: GitHub Actions, Docker Compose
- Логирование: ошибки и действия пользователей
**Ответственный агент:** docker-expert

### Детализация CI/CD и DevOps

**Docker и DevOps best practices:**
- Использование минимальных и безопасных базовых образов (Alpine, Distroless, Chainguard)
- Явное указание версий образов, отсутствие latest
- Разделение Dockerfile на слои: зависимости → исходники → сборка → запуск
- Хранение секретов вне образов (env, secrets, vault)
- Автоматизация сборки и деплоя через GitHub Actions
- Использование Docker Compose для локальной разработки и тестов
- Healthchecks для всех сервисов
- Логирование ошибок и действий пользователей
- Ежедневные бэкапы БД, хранение минимум 7 дней

**CI/CD pipeline:**
- Автоматическая сборка и тестирование при каждом push/PR
- Проверка стиля и линтинг (Go, JS, Docker)
- Сборка и публикация образов в registry
- Деплой на сервер/облако одной командой через Docker Compose
- Проверка прохождения всех тестов до деплоя
- Мониторинг состояния сервисов

**Критерии приёмки CI/CD:**
- Все сервисы запускаются одной командой (docker compose up)
- Нет секретов в репозитории и образах
- Все тесты проходят в CI
- Логи доступны для анализа
- Бэкапы и восстановление БД протестированы

**Ответственные агенты:**
- DevOps и Docker: docker-expert
- CI/CD pipeline: docker-expert, golang-expert, javascript-expert
- Code review: code-review-excellence

## 10. Критерии приёмки этапов и review-процедуры

### Общие принципы приёмки и review
- Для каждого этапа — обязательный чек-лист критериев и code review
- Code review — не только поиск ошибок, но и обмен знаниями, улучшение архитектуры и поддержка командной культуры
- Используются best practices из code-review-excellence (см. чек-листы и шаблоны ниже)
- Все критические изменения проходят review architecture-expert, security-expert, code-review-excellence
- Для крупных фич — предварительный дизайн-документ и review архитектуры
- Для каждого этапа — отдельный ответственный агент

### Процедура code review
1. **Контекст**: изучить описание задачи/PR, бизнес-требования, архитектурные решения
2. **High-level review**: архитектура, дизайн, тестовая стратегия, организация файлов
3. **Line-by-line review**: корректность, безопасность, производительность, поддерживаемость, тесты
4. **Summary & Decision**: резюмировать замечания, похвалить сильные стороны, вынести решение (approve, comment, request changes)
5. **Использовать чек-листы**: security, performance, testing (см. ниже)
6. **Соблюдать сроки**: review в течение 24 часов, не более 400 строк за раз
7. **Использовать шаблоны комментариев и severity-метки (🔴 blocking, 🟡 important, 🟢 nit, 💡 suggestion, 🎉 praise)**

### Чек-листы для review
- **Security**: валидация входных данных, параметризация SQL, отсутствие секретов, авторизация, XSS/CSRF, rate limiting
- **Performance**: отсутствие N+1, индексация, пагинация, кэширование, отсутствие блокирующих операций
- **Testing**: покрытие happy path, edge cases, error cases, детерминированность тестов

- Зависимость от внешних сервисов (Госуслуги, RuStore, Google)

- Архитектурная изоляция интеграций (Hexagonal, DDD)
- Регулярные бэкапы и тестирование восстановления
- Мониторинг и алерты на ключевые метрики
- Документирование всех интеграций и ограничений
